# DESIGN.md

このドキュメントは、このリポジトリにおける **UI デザインの唯一の基準（Source of Truth）** です。  
画面実装を追加・変更する場合、まず本書を確認し、既存画面との整合を崩さないこと。

---

## 1. デザイン哲学（Design Philosophy）

- 目標は **落ち着き・信頼感・運用性**。
- 派手な演出より、業務画面としての **可読性 / 視認性 / 誤操作防止** を優先。
- 管理画面・社内システム・B2B SaaS のどれにも適用できる汎用トーンを維持。
- 「情報を早く正確に読む」「迷わず入力する」「状態変化を誤解しない」を最重要指標とする。

---

## 2. 日本語タイポグラフィ

### 2.1 フォントスタック

基本は日本語優先サンセリフを使う。

```css
font-family: "Inter", "Hiragino Kaku Gothic ProN", "Hiragino Sans", "Yu Gothic UI",
  "Yu Gothic", "Meiryo", "Noto Sans JP", system-ui, -apple-system, sans-serif;
```

- 見出し・本文・フォームラベルを同一系統で統一し、運用画面のノイズを減らす。
- コードや ID は等幅フォント（`ui-monospace` 系）を使う。

### 2.2 本文ルール

- 本文サイズ: `14px`〜`16px` を標準。
- 行間: 日本語本文は `line-height: 1.65`〜`1.8`。
- 字間: `letter-spacing: 0`〜`0.02em`（詰めすぎ禁止）。
- 1段落は 2〜4 文で分け、長文を避ける。

### 2.3 見出しルール

- `h1` は密度を下げ、`line-height: 1.35`〜`1.5`。
- `h2` 以下は本文との間に十分な余白（24px 以上）を確保。
- 英数字主体の短い識別ラベル（例: ID、SKU）は別行にせず、本文文脈に添える。

### 2.4 日本語特有の運用ルール

- 句読点・括弧の直前直後で視覚的に詰まりすぎないよう、短い文で区切る。
- 全角/半角混在（例: 「ID 1234」「2026/04/19」）は、ラベルと値の間に半角スペースを入れる。
- 長い日本語ラベルは無理に 1 行固定しない。折り返しを許容し、`line-height` を確保する。

## 3. カラーシステム

### 3.1 基本方針

- 信頼感のある青〜青灰系を主軸にする。
- 背景は白〜ごく薄いグレー。境界線は低コントラストのグレーで静かに区切る。
- 強いグラデーションや高彩度アクセントは使用しない。

### 3.2 基本トークン（推奨）

- Canvas 背景: `#f7f9fc`
- Surface（カード）: `#ffffff`
- Surface muted: `#f2f5fb`
- Text strong: `#1f2a37`
- Text body: `#334155`
- Text muted: `#5b6777`
- Border default: `#d7dfeb`
- Border soft: `#e4e9f2`
- Primary: `#2e5ea8`（hover: `#244e8f`）

### 3.3 セマンティックカラー

- Success: `#1f7a52`
- Warning: `#9a6700`
- Danger: `#c0392b`
- Info: `#1f5f99`

運用:
- 成功/警告/エラーは **テキスト + アイコン/ラベル** で示し、色だけに依存しない。
- エラー色の塗り面積は最小限（枠線・ラベル・補助文中心）。

---

## 4. 余白とレイアウト

### 4.1 スペーシングスケール

- 8px ベース（必要に応じて 4px 補助）。
- 主要ステップ: `4, 8, 12, 16, 24, 32, 40, 48, 64`。

### 4.2 ページレイアウト

- 標準ページ幅: `max-width: 1200px`（読み物中心は `760`〜`920px`）。
- 画面外周余白: モバイル `16`〜`24px`、デスクトップ `24`〜`40px`。
- セクション間余白: 最低 `24px`、重要ブロック間は `32px` 以上。

### 4.3 カード・フォーム・テーブル

- カード内 padding: `16`〜`24px`（大枠カードは `32px` まで可）。
- フォーム縦間隔: ラベル〜入力 `6`〜`8px`、項目間 `16`〜`24px`。
- テーブル: 行高を確保し、詰め込みすぎない（目安 44px 以上）。

---

## 5. コンポーネントルール

### 5.1 Buttons

- 角丸: `8px` 前後。
- 主要ボタンは Primary 塗り + 白文字、二次ボタンは白背景 + 境界線。
- 高さは `36`〜`40px` 目安。

### 5.2 Inputs / Select / Textarea

- 背景は白、境界は `border-default`。
- フォーカス時は Primary 系アウトラインを明示。
- プレースホルダは薄すぎない muted 色。

### 5.3 Cards

- 白背景 + 細い境界線 + 弱い影。
- 角丸は `8`〜`12px`。

### 5.4 Tables

- ヘッダーと本文のコントラストを明確化。
- 横罫線は薄く、縦罫線は極力減らす。

### 5.5 Empty states

- 何が空か / 次に何をするかを短く明示。
- 感情的な演出より実務行動（例: 「条件を変更」「新規作成」）を優先。

### 5.6 Section headers

- タイトル + 説明文（1〜2文）をセットで配置。
- 説明文は誤操作防止に寄与する情報を優先。

### 5.7 Status badges

- 色 + 短い語句で状態を明示（例: 「承認済み」「確認待ち」）。
- 彩度を抑え、本文より目立ちすぎない。

### 5.8 Validation messages

- 入力欄の近くに表示し、何を修正すべきかを具体的に書く。
- 口調は簡潔・事務的に。

### 5.9 Page titles / descriptions

- ページ上部に「タイトル」「目的説明」「主操作」を順に置く。
- タイトルだけで終わらせず、運用上の文脈を 1 文添える。

---

## 6. インタラクションルール

- hover: 色差は小さく（軽い濃淡変化）。
- focus: キーボード操作で必ず視認可能な `focus-visible` を表示。
- active: 押下感は軽微（暗くしすぎない）。
- error: 赤一色で威圧しない。原因と対処をセットで示す。
- loading: スピナーのみではなく、処理対象を簡潔に文言で示す。

---

## 7. コンテンツスタイル（文言トーン）

- 日本語 B2B / 管理業務向けの丁寧で簡潔なトーン。
- ラベルは短く、意味が一意に伝わる語彙を使う。
- 補助文は「何を入力すると安全か」「どの条件で確定するか」を優先。
- エンプティ/エラー文言は実務的に（かわいい演出・過剰な比喩は禁止）。
- マーケティング調コピーより、運用判断に必要な情報を先に置く。

---

## 8. DO / DON’T

### DO

- 十分な行間と余白で、日本語の可読性を担保する。
- Primary 色は行動誘導に限定し、常時多用しない。
- カード・フォーム・テーブルの密度を合わせ、画面ごとの差を減らす。
- エラーは「原因 + 修正方法」を明示する。

### DON’T

- 過度なグラデーション、強いシャドウ、ビビッドな多色構成。
- 英語 UI 前提のまま和文を詰め込む（行間不足・長文 1 行固定）。
- 色だけで状態を伝える。
- 装飾的な空状態メッセージで次アクションが不明瞭なままにする。

---

## 9. 実装運用ルール（このリポジトリ向け）

- 新規 UI 実装時は、まず `app/globals.css` のデザイントークンを流用する。
- 既存画面改修は最小差分で行い、トーン統一を優先する。
- 業務ロジックや DB 構造の変更を伴う場合は、本書ではなく `AGENT.md` のアーキテクチャ規約を優先する。
